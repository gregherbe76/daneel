ÉTAPE 2 — IMPLÉMENTATION (après mon OK)

A. Définir EvaluationProvider interface (si pas déjà présente)

   - Dans interface.ts ou un nouveau fichier evaluation-interface.ts
   - Méthode evaluate(input) → EvaluationResult
   - input: { candidate: { github_username?: string, ...other fields } }
   - EvaluationResult :
     {
       evaluated: boolean,
       provider_name: string,
       scores: {
         technical_depth: number,
         ownership: number,
         consistency: number,
         taste: number,
         impact: number,
         overall: number
       } | null,
       strengths: string[],
       red_flags: string[],
       summary: string,
       evaluated_at: string,
       report_url: string | null,
       error: string | null
     }
   - Si le candidat n'a pas de github_username : retourne 
     { evaluated: false, error: "no_github_username", ... }

B. Ajouter l'étape "evaluation" au workflow engine

   - Étape OPTIONNELLE entre candidate_matching et 
     shortlist_generation (ou en parallèle, selon ce qui fait sens 
     dans l'engine actuel)
   - Doit pouvoir s'activer/se désactiver par job
   - Quand activée, appelle le provider d'évaluation configuré pour 
     chaque candidat avec un github_username
   - Stocke le résultat dans la table candidates (nouveau champ 
     evaluation_result JSON ou nouvelle table candidate_evaluations 
     selon le pattern existant — préfère le pattern le plus simple)

C. Créer codematch-provider.ts

   - artifacts/api-server/src/routes/workflows/providers/codematch-provider.ts
   - Implémente EvaluationProvider
   - Lit la config provider depuis la BDD : api_key (chiffré), 
     api_url (default https://assess.codes/api/v1)
   - Méthode evaluate :
     1. Si pas de github_username → retourne early avec evaluated:false
     2. Appelle POST {api_url}/evaluate avec Bearer {api_key} et 
        body { github_username }
     3. Gère les erreurs HTTP :
        - 401/403 → log "CodeMatch API key invalid", retourne 
          evaluated:false, error:"auth_failed"
        - 402 → log "CodeMatch user not Premium", retourne 
          evaluated:false, error:"premium_required"
        - 404 → retourne evaluated:false, error:"github_user_not_found"
        - 429 → log rate limit, retourne evaluated:false, 
          error:"rate_limited"
        - 5xx → retourne evaluated:false, error:"server_error"
        - timeout 30s → retourne evaluated:false, error:"timeout"
     4. Si 200 : map la réponse JSON CodeMatch vers EvaluationResult
     5. Headers à respecter : User-Agent: "Daneel/1.0 (CodeMatch 
        provider)"

D. Enregistrer codematch dans la registry

   - registry.ts : ajouter "codematch" comme provider type pour 
     l'étape "evaluation"
   - Créer un default config si nécessaire
