ÉTAT EXTEND API CONFIRMÉ

Endpoints publics (Bearer ex_xxx, Premium uniquement) :
- POST /api/v1/find-similar
  Body : { profile_urls: string[] (1-10), job_context?, 
           search_location?, max_results?, min_results?, 
           scoring_criteria? }
  Retour : 202 + { analysis_id, status: "running", poll_url, 
           created_at }
- GET /api/v1/find-similar/:id
  Retour : 200 + { status: "running" | "completed" | "failed", ... }
  Si completed : reference_profile_urls, patterns, stats, 
  profile_syntheses, candidates, total_found, below_minimum, 
  is_refined, refined_from_id
- POST /api/v1/refine
  Body : { analysis_id, feedback: [{candidate_url, verdict, 
           comment?}] }
  Retour : 202 + { analysis_id (nouveau), refined_from_id, 
           poll_url, status: "running" }

Cookie LinkedIn géré côté Extend (pas dans le body), config 
strictement BDD en prod (fix sécurité).

DÉCISIONS DESIGN ARBITRÉES

D1. Type provider : "extend" ajouté au providerTypeEnum, suit 
    le contrat AgentProvider générique (PAS d'interface dédiée 
    EvaluationProvider — Extend reste un sourcing provider 
    classique avec input job + résultat candidates)

D2. Profile URLs comme input job-level :
    - Nouvelle colonne jobs.exampleProfileUrls : text[] nullable
    - Page config job : nouveau champ "Example profiles 
      (for Extend)" multi-input (1 à 10 URLs)
    - Si Extend sélectionné comme sourcing provider mais 
      exampleProfileUrls vide : workflow échoue avec message 
      "Extend requires example profile URLs"

D3. Refinement loop : SKIP pour la v1 de l'intégration
    - On expose seulement POST /v1/find-similar + polling
    - Le refinement loop d'Extend reste accessible via Extend 
      standalone uniquement
    - Documentation : "For pattern refinement based on candidate 
      feedback, use Extend directly at extend.aplayer.ai"
    - Justification : intégrer le refinement dans Daneel demande 
      une UX multi-tours qui dépasse le scope. À ré-évaluer en 
      Sous-tâche 3.4 si signal utilisateur

D4. Async polling côté Daneel :
    - Le provider Extend appelle POST /v1/find-similar (202 + 
      analysis_id)
    - Puis polle GET /v1/find-similar/:id toutes les 10 secondes 
      jusqu'à status "completed" ou "failed"
    - Timeout côté Daneel : 12 minutes (Extend a un watchdog 10 
      min côté serveur, on prend une marge)
    - Si timeout : retourner sourced: false, error: "timeout"

D5. Mapping candidates Extend → format Daneel SourcingCandidate
    - Extend retourne candidates avec : { name, linkedinUrl, 
      score, scoreReason, hiringSignal?, ... }
    - Daneel SourcingCandidate attend : { name, headline, 
      location, currentCompany, email, linkedinUrl, githubUrl, 
      username, confidence, emailSource, skills, summary, 
      evidence, potentialRisks, source }
    - Mapping :
      - name → name (direct)
      - linkedinUrl → linkedinUrl (direct, mais camelCase Extend → 
        snake_case Daneel selon la convention Daneel)
      - score (0-10 chez Extend) → confidence (0-1 chez Daneel) 
        après /10
      - scoreReason → summary (concaténation des bullets)
      - source → "Extend pattern-match"
      - email, githubUrl, username : null (Extend ne les fournit 
        pas)
      - emailSource : null
      - skills : extraire depuis le pattern si possible, sinon []
      - evidence : linkedinUrl
      - potentialRisks : "" (Extend ne flag pas explicitement)
      - headline, location, currentCompany : extraire depuis 
        scoreReason si parsable, sinon ""
